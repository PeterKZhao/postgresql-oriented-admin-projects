# 多语言化模型卡

需求来源：`requirements.txt` 的“景区官网 / 多语言化”。

模型类型：内容数据模型 + 语言回退规则。该需求本身不是金额、容量、路线类数学模型，但需要明确状态、唯一性、不变量和异常处理。

参考能力：ruoyi-vue-pro 前端支持 i18n，可用于 UI 文案、菜单、按钮、表单提示等界面语言切换。第一版按用户决策，后端多语言优先使用 ruoyi-vue-pro 支持，不默认新增自定义 `*_i18n` 表。

## 参与者

- 游客：在官网、小程序或 H5 中选择中文/英文浏览内容。
- 内容编辑人员：维护中文内容和英文翻译内容。
- 翻译/审核人员：提交、审核、发布翻译。
- 景区运营人员：配置默认语言、可用语言、语言回退规则。
- 系统管理员：维护语言字典、菜单、权限、基础配置。
- 前端应用：根据当前语言渲染 UI 和业务内容。
- 后端服务：按语言查询内容，执行回退和发布状态过滤。

## 输入

- 当前请求语言：如 `zh-CN`、`en`。
- 默认语言：第一版建议 `zh-CN`。
- 回退语言规则：第一版建议 `en -> zh-CN`。
- 内容主体编号：文章、景区、场馆、活动、线路、POI、广告等业务实体 ID。
- 内容类型：`site_article`、`tourism_scenic`、`tourism_activity`、`map_poi` 等。
- 多语言字段：标题、简介、正文、SEO 标题、SEO 描述、图片说明、视频说明等。
- 翻译状态：草稿、待审核、已发布、已下线。
- 发布时间、下线时间、更新时间。
- 租户编号：如系统启用多租户，内容翻译必须隔离租户。

## 输出

- 指定语言下的页面 UI 文案。
- 指定语言下的业务内容。
- 缺失翻译时的回退内容。
- 内容翻译完整度：已翻译字段数、缺失字段数、缺失语言列表。
- 后台翻译管理列表：按内容、语言、状态、更新时间筛选。
- 前端 SEO 元数据：标题、描述、关键词。

## 状态

- 语言状态：启用、禁用。
- 翻译状态：
  - `draft`：草稿，仅后台可见。
  - `pending`：待审核。
  - `published`：已发布，前台可见。
  - `offline`：已下线，前台不可见。
- 内容主体状态：启用、禁用、删除。
- 回退状态：
  - 命中目标语言。
  - 目标语言缺失，命中默认语言。
  - 目标语言和默认语言都缺失，返回不可见或空内容。

## 金额/次数/时间单位

- 金额：无。
- 次数：
  - 翻译字段数量：`int`。
  - 缺失字段数量：`int`。
  - 语言数量：`int`。
- 时间：
  - 发布时间：`LocalDateTime`。
  - 下线时间：`LocalDateTime`。
  - 更新时间：`LocalDateTime`。
- 语言编码：使用字符串，如 `zh-CN`、`en`，建议统一走字典或枚举校验。

## 核心规则

1. UI 文案优先使用 ruoyi-vue-pro 前端 i18n 能力。
2. 第一版业务内容多语言优先使用 ruoyi-vue-pro 后端支持，不默认新增自定义翻译表。
3. 每个可多语言展示的业务主体，至少要有默认语言内容。
4. 前台只展示 `published` 状态的翻译内容。
5. 当目标语言内容不存在或未发布时，按回退规则读取默认语言。
6. 回退内容必须明确标识，便于前端或日志知道当前展示的不是目标语言。
7. SEO 字段需要随语言切换，不能只翻译正文。
8. 图片、视频、地图点位说明等媒体附属文案也应支持多语言。
9. UGC 用户内容默认不自动翻译；如运营精选展示到官网，可单独维护官方翻译。
10. 多租户场景下，不同租户的翻译内容不能互相读取。

## 不变量

- 同一租户、同一业务主体、同一语言只能有一条有效翻译记录。
- 已发布翻译必须关联存在且未删除的业务主体。
- 已发布翻译的标题不能为空。
- 默认语言内容不能为空，否则该业务主体不能前台发布。
- 请求语言必须属于启用语言集合。
- 禁用语言不能在前台被选中。
- 删除业务主体时，对应翻译不能继续前台展示。
- 前台不能展示 `draft`、`pending`、`offline` 状态的翻译。

## 异常场景

- 游客选择英文，但该内容没有英文翻译：回退到中文，并记录回退结果。
- 英文翻译只填写标题，正文缺失：按字段级策略决定整条回退或缺失字段回退；第一版建议整条回退，避免中英混排。
- 翻译已下线：前台回退到默认语言。
- 默认语言也缺失：前台不展示该内容，后台提示必须补齐默认语言。
- 请求语言非法：回退到默认语言。
- 源内容被删除：所有翻译前台不可见。
- 缓存未刷新：发布、下线、删除翻译后必须清理内容缓存。
- 多租户串数据：查询必须带 `tenant_id`。

## 复用 ruoyi 表

- `system_dict_type`、`system_dict_data`：可维护语言编码、翻译状态等字典。
- `infra_config`：可保存默认语言、启用语言、回退策略等配置。
- `infra_file`：复用图片、视频、附件存储。
- `system_menu`：后台菜单名称可结合 ruoyi 前端 i18n 处理。
- ruoyi-vue-pro 前端 i18n：复用 UI 文案国际化能力。
- `promotion_banner`、`promotion_article`、`promotion_diy_page`：如使用 mall/promotion 文章、banner、装修能力，需要补充对应翻译扩展。

## 需要新增的扩展表

第一版不新增自定义 `*_i18n` 表。

原因：

- 前端 UI 文案使用 i18n 模块。
- 后端业务内容多语言优先使用 ruoyi-vue-pro 支持。
- 不为本地活动单独创建自定义多语言扩展表。

保留风险：

- 如果后续验证 ruoyi-vue-pro 无法覆盖某些业务内容字段的多语言存储，再单独评估是否增加具体扩展表。
- 若需要新增，也应按业务对象逐表评估，不默认创建全量 `*_i18n` 表。

## Java 服务建议

- 优先接入 ruoyi-vue-pro 已有 i18n 能力，不先自研完整翻译服务。
- `LocaleResolveService`：如 ruoyi 现有能力不足，再封装请求语言、默认语言和回退语言解析。
- `I18nCacheService`：如多语言内容需要缓存，再处理发布后的缓存失效。

实现建议：

- 前台查询服务和后台管理服务分开，避免前台读到草稿。
- 回退策略集中处理，不要散落在 Controller 或 Mapper。
- 不新增 `*_i18n` 表时，需要明确 ruoyi-vue-pro 支持的字段范围和回退行为。

## 测试样例

1. 请求 `zh-CN`，中文内容已发布，应返回中文内容，`fallbackUsed=false`。
2. 请求 `en`，英文内容已发布，应返回英文内容，`fallbackUsed=false`。
3. 请求 `en`，英文内容不存在，中文内容已发布，应返回中文内容，`fallbackUsed=true`。
4. 请求 `en`，英文内容是 `draft`，中文内容已发布，应返回中文内容。
5. 请求非法语言 `fr-FR`，应回退到默认语言 `zh-CN`。
6. 默认语言内容不存在，业务主体不能发布到前台。
7. 同一主体同一语言重复新增翻译，应被唯一约束或服务校验拒绝。
8. 翻译发布后，前台缓存应刷新并返回最新内容。
9. 源内容删除后，对应翻译不能继续展示。
10. 不同租户下相同主体编号不能读到对方翻译。
